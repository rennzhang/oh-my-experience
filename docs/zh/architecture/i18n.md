# 国际化


## 范围

国际化有四个表面：

- CLI 显示语言；
- 经验卡内容语言；
- 跨语言召回匹配；
- 公开文档语言。

## 运行时语言策略

语言不写入 `config.json`。

- CLI 人类输出默认英文，以符合国际开源工具的预期。
- `OME_LANGUAGE=zh-CN` 是显式中文覆盖。
- JSON 输出字段名保持稳定英文。
- 经验卡语言由 reflect 产物自然决定。
- Hook context 使用固定英文框架包裹已存储的卡片正文。

## CLI

CLI help 和人类可读结果使用 message key，默认英文。

第一批支持语言是英文和简体中文。

## 经验卡语言

每张卡片都应该声明内容语言。生成经验时，默认使用最贴近来源会话和用户工作
习惯的语言；命令、包名、路径、API 和技术术语保持原文。首装向导不暴露这个
配置。

Hook 注入给 Agent 的框架提示固定使用英文。卡片正文保持原语言，避免翻译损失
语义细节。

## 召回语言

召回引擎必须支持跨语言召回，但不能在 hook 热路径中做翻译。

推荐做法：

- reflect candidate creation 阶段生成双语触发词；
- 每张卡片允许维护 aliases；
- 保留命令、包名、路径和代码 token；
- 使用中文短语和 bigram 匹配；
- 在 `ome eval recall` 中评估双语 fixture。

## 文档语言

面向用户的公开文档使用路径式 locale：

- 英文：`/`
- 简体中文：`/zh/`

每种语言保持同构目录：

```text
docs/guides/quickstart.md
docs/zh/guides/quickstart.md
```

未来新增语言时，新增一个顶层 locale 目录，并在
`docs/.vitepress/config.ts` 的 `locales` 中增加对应配置。不要再创建
`quickstart.zh-CN.md` 这类语言后缀路由。

命令、路径、配置键、frontmatter 字段、API 字段和包名不能翻译。应该翻译它们周围的说明文字。

原始来源归档和用户原话记录可以保持原语言，但公开入口应该链接到对应语言
版本。
